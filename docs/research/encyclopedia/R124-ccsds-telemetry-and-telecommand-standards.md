## 1. Purpose

`spacesim` models command uplink as an abstract "validate → window → execute" sequence
([R103](R103-satellite-command-and-control.md)) without simulating a wire protocol. This topic gives the implementer the
real CCSDS standard underneath that abstraction — the actual telecommand frame structure and
retransmission protocol every real spacecraft command uses — so a future higher-fidelity feature
(e.g. a simulated command-rejection-due-to-frame-loss mechanic) extends the model with the real
protocol's actual failure modes rather than an invented one.

## 2. Scope

Covers: the CCSDS TC Space Data Link Protocol's frame structure, the COP-1 retransmission/sequence-
control procedure, and command-counter-based acknowledgment. Does **not** cover: the application-
layer command validation logic that decides *whether* a command is allowed
([R103](R103-satellite-command-and-control.md)/[R114](R114-command-and-data-handling.md)), or the console software that constructs these frames from a
command database ([R123](R123-command-and-telemetry-console-software.md)).

## 3. Concepts

**Real telecommands are sent as CCSDS Transfer Frames, not raw application data.** The CCSDS
TC Space Data Link Protocol standard defines the forward-link (ground-to-spacecraft) frame format
used to send commands, distinguishing three frame types: **AD frames** (normal telecommands subject
to sequence-controlled transmission verification), **BD frames** (normal telecommands *not* subject
to that verification), and **BC frames** (control commands generated by the ground COP-1 state
machine to control the onboard state machine)
([CCSDS, *TC Space Data Link Protocol*, CCSDS 232.0-B-4](https://ccsds.org/Pubs/232x0b4e1c1.pdf)
([Wayback](https://web.archive.org/web/2026/https://ccsds.org/Pubs/232x0b4e1c1.pdf))) — `spacesim`'s
`OrderSystem` command path corresponds conceptually to the AD-frame case: every command the engine
schedules is implicitly "verified" delivery (it either executes at the window or doesn't), with no
modeled equivalent of a BD-frame fire-and-forget command.

**COP-1 is the one standardized retransmission procedure, built on a sliding-window sequence
count.** Only one Communications Operation Procedure is defined in the standard, named COP-1, whose
purpose is "to provide a sequence-controlled transmission... which ensures a high probability of
success" for AD-frame commands
([CCSDS, *Telecommand Part 2.1 — Command Operation Procedures*, CCSDS 202.1-B-2](https://ccsds.org/Pubs/202x1b2s.pdf)
([Wayback](https://web.archive.org/web/2026/https://ccsds.org/Pubs/202x1b2s.pdf))) — real ground
software (the T&C systems documented in [R123](R123-command-and-telemetry-console-software.md)) retransmits a command frame if the
expected acknowledgment doesn't arrive within the protocol's window, rather than firing a command
once and trusting it landed; `spacesim` does not model this retry layer at all — a command either
finds (and uses) a valid access window or it doesn't, with no modeled frame-loss/retransmit case.

**A monotonic command counter, not a content hash, is the real acknowledgment mechanism.** The
spacecraft regularly downlinks a status word containing "the value of the latest received command
counter and a few status bits" so ground software can confirm which commands the bus has actually
received and processed (per CCSDS's COP-1 sliding-window design, *op. cit.*) — this is the real-world
analog of `spacesim`'s simpler "did the scheduled event fire" confirmation, but the real mechanism
is link-level (a counter the bus itself increments) rather than an event-log entry the ground system
trusts unconditionally.

## 4. Operational Context

Every real spacecraft command, regardless of mission or vendor, ultimately travels as a CCSDS TC
transfer frame governed by COP-1's sequence-controlled delivery — this near-universal adoption is
why the real T&C systems in [R123](R123-command-and-telemetry-console-software.md) can claim cross-mission, cross-vendor interoperability:
the protocol layer is standardized even though the command catalog above it is mission-specific.
`spacesim`'s abstraction (a command either executes at a valid window or doesn't, with the engine
itself as the implicit acknowledgment authority) is a deliberate simplification of a real link layer
that includes its own retransmission and counter-based confirmation.

## 5. Implementation Guidance

- **Do not invent a frame-loss or retransmission mechanic without grounding it in COP-1's actual
  sliding-window design** — if a future fidelity tier wants to simulate command-delivery failure, it
  should model something resembling AD-frame sequence-controlled transmission (a command can be
  un-acknowledged and require retransmission within the same window), not an arbitrary random-failure
  roll.
- **Keep the command-counter analogy at the abstraction level `spacesim` already uses** — the engine's
  event log already plays the role of "what the bus actually received," so a new feature should not
  add a second, parallel acknowledgment mechanism that could disagree with the event log.
- **A new command-validation feature belongs in `OrderSystem`'s `_validate`, not in a simulated
  frame/protocol layer** — this topic documents the real protocol for grounding purposes; `spacesim`
  deliberately does not implement CCSDS framing itself, and a new feature should not start doing so
  without an explicit fidelity-tier decision.

## 6. Feature Mapping

FS-105 (Spacecraft Operations) is the indirect consumer — this topic exists to ground any future

no current feature requires implementing the protocol itself.
